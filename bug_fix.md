# 🏆 Codeforces to Baekjoon (v3.12 Query Engine & Pagination Master)

본 명세서는 한 화면에 렌더링되는 문제 수를 유저가 직접 제어할 수 있는 동적 페이지네이션(Dynamic Pagination) 기능과, solved.ac의 강력한 문법을 코드포스 생태계에 맞게 개조한 복합 쿼리 파서(AST Query Parser)의 도입 지침을 정의합니다.

---

## 🚨 1. 3회 교차 검증 및 아키텍처 분석 (3-Pass Debugging)

기능을 설계하며 발생할 수 있는 치명적 오류를 스스로 3회 검증하고 방어 로직을 구축했습니다.

### ① 1차 검증 (페이지네이션 충돌): "5페이지에서 20개 보기로 바꾸면 어떻게 되는가?"
* **위험 요소:** 유저가 100개씩 보기 모드에서 5페이지(401~500번 문제)를 보다가, 20개씩 보기 모드로 셀렉트 박스(Toggle)를 변경할 경우, 전체 배열 인덱스가 꼬여 빈 화면이 출력되거나 앱이 다운될 수 있습니다.
* **방어 설계:** '페이지당 문제 수(Items per page)'가 변경되는 이벤트가 발생할 때마다, 현재 뷰를 강제로 1페이지(`currentPage = 1`)로 초기화하는 상태 롤백(State Rollback) 방어벽을 컨트롤러 최상단에 세워야 합니다.

### ② 2차 검증 (정규식 파괴): "OR 연산자(|)와 AND 연산자(공백)의 혼용"
* **위험 요소:** 기존 엔진은 무조건 띄어쓰기 기준으로 알약을 쪼갰습니다. 만약 유저가 `#dp | #math` (DP 이거나 수학)라고 치면 파서가 붕괴됩니다.
* **방어 설계:** 파서의 평가(Evaluator) 로직을 2-Depth로 쪼갭니다. 먼저 입력된 전체 쿼리를 `|` (OR 연산자) 기준으로 그룹화하여 나눈 뒤, 각 그룹 내부를 공백(AND 연산자)으로 나눕니다. 즉, "A그룹을 모두 만족하거나, B그룹을 모두 만족하면 통과"시키는 다중 논리 회로를 구축합니다.

### ③ 3차 검증 (범위 연산자 오류): "1500점 이상 2000점 이하를 어떻게 캘 것인가?"
* **위험 요소:** solved.ac의 핵심인 `..` (Range) 연산자를 문자열 비교(`includes`) 방식으로 처리할 수 없습니다.
* **방어 설계:** 파서 내부에 범위 해석기(Range Extractor)를 도입합니다. 쿼리 문자열에 `..`이 포함되어 있다면 앞뒤의 숫자를 파싱하여 `min`과 `max` 변수로 치환한 뒤, 문제의 `rating`이나 `solvedCount`가 이 숫자 사이에 들어오는지 부등호(`>=`, `<=`)로 수학적 검증을 하도록 엔진을 업그레이드합니다.

## 🔍 2. Codeforces 맞춤형 solved.ac 쿼리 사전 (Query Dictionary)

코드포스 API로 가져올 수 없는 데이터(언어, 기여자 등)는 제외하고, 완벽하게 호환되는 핵심 쿼리들만 추려내어 정의했습니다.

| 쿼리 타입 | solved.ac 문법 | 코드포스(v3.12) 호환 문법 및 동작 원리 |
| :--- | :--- | :--- |
| **태그** | `#dp`, `tag:dp` | `#dp` 또는 `tag:dp` 입력 시 `tags` 배열과 대조. (언더바 `_` 입력 시 공백으로 자동 치환) |
| **티어(레이팅)** | `t:s3`, `tier:g..p` | **`r:` 또는 `rating:` 도입.** <br>`r:1500` (1500점 정확히 일치) <br>`r:1500..2000` (1500점 이상 2000점 이하) <br>`r:..1200` (1200점 이하) <br>`r:3000..` (3000점 이상) |
| **해결 여부** | `@handle`, `!@handle` | **타인 조회 불가 특성 반영.** <br>`@me` (내가 맞힌 문제) <br>`~@me` 또는 `!@me` (내가 안 푼 문제) <br>`?@me` (시도했으나 틀린/미해결 문제) |
| **정답자 수** | `s#100..` | **`s:` 또는 `solved:` 도입.** <br>`s:1000..` (정답자가 1000명 이상인 문제) <br>`s:..50` (정답자가 50명 이하인 문제) |
| **문제 번호** | `id:1000` | **`id:` 또는 `c:`(대회), `i:`(인덱스) 도입.** <br>`id:2220B`, `c:2220` (2220번 대회 묶음), `i:A` (A번 문제 묶음) |

## 🛠️ 3. 아키텍처 개편 및 조치 지침 (Implementation Guide)

### Step 1. 동적 페이지네이션 UI 및 로직 구축
1. **셀렉트 박스(UI) 주입:** 통합 검색창(`<div class="boj-search-input-wrapper">`) 우측 끝이나 바로 아래에, 한 페이지에 보일 문제 수를 선택할 수 있는 `<select>` 드롭다운 메뉴(옵션: 20개, 50개, 100개)를 생성하십시오.
2. **전역 설정(Config) 동기화:** 해당 셀렉트 박스의 값이 변경될 때마다(Change Event), 기존에 하드코딩했던 `Config.MAX_RENDER_COUNT` 값을 유저가 선택한 숫자로 덮어씌우십시오.
3. **상태 초기화(Rollback):** 셀렉트 박스 값이 변경되면 즉시 `currentPage` 변수를 `1`로 초기화하고 가상 테이블 렌더링 함수를 재호출하십시오.

### Step 2. 범위(Range) 연산자 엔진 탑재
1. **파서 모듈(`queryParser.js`) 개조:** `evaluateToken` 함수 내부를 전면 개편합니다.
2. `r:`(레이팅) 또는 `s:`(정답자 수) 접두사로 시작하는 토큰을 만났을 때, 문자열 내에 `..` 이 있는지 검사하는 조건문을 추가하십시오.
3. `..` 이 존재한다면 `split('..')`을 통해 좌측 값을 `min`, 우측 값을 `max` 변수에 담으십시오. (좌측이 비어있으면 `min = 0`, 우측이 비어있으면 `max = Infinity`로 처리)
4. 해당 문제의 실제 `rating` 또는 `solvedCount` 값이 `min`과 `max` 사이에 존재하는지(`min <= value && value <= max`) 확인하여 true/false를 반환하도록 수학 논리를 작성하십시오.

### Step 3. 복합 논리 연산자(OR, AND) 처리
1. 통합 검색창에서 엔터를 쳤을 때, 더 이상 띄어쓰기로 무지성 분리(Tokenization)를 하지 마십시오.
2. 유저가 입력한 전체 텍스트를 그대로 하나의 거대한 '알약(Pill)'으로 만드십시오. (예: `#dp | r:1500..2000`)
3. 가상 테이블 렌더링을 위해 파서가 알약을 해독할 때, 먼저 `|` 문자를 기준으로 문자열을 쪼개어 **'OR 배열'**을 만드십시오.
4. 그 다음, 각각의 OR 배열 요소 내부를 공백(띄어쓰기)으로 쪼개어 **'AND 배열'**을 만드십시오.
5. 최종적으로 "OR 배열 중 단 하나라도(some), 내부의 AND 배열 조건을 모두(every) 만족하는가?"를 판별하는 이중 반복문으로 평가(Evaluator) 로직을 업그레이드하십시오.
