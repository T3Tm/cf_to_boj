# 🏆 Codeforces to Baekjoon (v3.9 Integrity & Global Theme Master)

본 명세서는 아키텍처 내부에 잔존하는 하드코딩(Hardcoding) 요소를 완전히 발본색원하고, 특정 페이지에 국한되었던 '백준 스타일의 카드 레이아웃'을 모든 페이지(전역)로 확장하기 위한 최종 설계 지침서입니다.

## 🚨 1. 3회 교차 검증 및 아키텍처 분석 (Verification)

### ① 1차 검증: `getUserTierIcon`은 왜 대충 하드코딩 되었는가?
* **원인:** 이전 업데이트에서 수학적 계산식(`Math.floor`)이 유발하는 404 에러를 막기 위해 명시적 조건문으로 구조를 바꿨습니다. 이때 문제 티어 계산기는 100점 단위로 촘촘히 짰으나, 유저 티어 계산기는 코드포스의 넓은 유저 랭크 구간(예: 1200~1399 Pupil)을 백준의 5단계 티어에 대충 욱여넣은 '임시 예시 로직'이 그대로 릴리스되었습니다.
* **결과:** 1200점 유저와 1390점 유저가 모두 똑같은 '실버 3'으로 표기되는 등 유저 레이팅의 변별력이 완전히 사라졌습니다.
* **해결 설계:** 유저 티어 역시 문제 티어와 동일하게 100점 단위로 세밀하게 쪼개어, 각 점수대마다 1부터 5까지의 명확한 SVG 파일명을 반환하도록 조건문을 20개 이상으로 촘촘하게 전면 재작성해야 합니다.

### ② 2차 검증: 다른 곳에 숨겨진 하드코딩은 없는가? (Magic Numbers)
* **원인 및 발견:** `problemset.js`의 가상 테이블 렌더링 로직을 검토한 결과, 브라우저 렌더링 과부하를 막기 위해 배열을 자르는 부분에 `100`이라는 숫자가 생으로 박혀있습니다. 또한 프로필 페이지 등에서 폭을 제한하는 `1000px` 같은 수치도 하드코딩되어 있습니다.
* **해결 설계:** 이러한 숫자들은 '매직 넘버(Magic Number)'입니다. 파일 최상단에 환경설정(Config) 상수 객체를 만들고, `MAX_RENDER_COUNT = 100`과 같이 이름을 부여하여 관리해야 합니다. 추후 사용자가 "한 페이지에 50개만 보고 싶다"고 할 때 로직을 뒤지지 않고 상단 변수만 수정할 수 있도록 유연하게 개조해야 합니다.

### ③ 3차 검증: 백준 느낌이 '모든' 페이지에 적용되었는가?
* **원인:** 이전 업데이트에서 사이드바를 지우고 중앙 정렬을 하는 로직을 `problem.js`와 `problemset.js` 컨트롤러 내부에 각각 개별적으로 작성했습니다. 그 결과, 채점 현황(`status`)이나 카탈로그(`catalog`), 그룹(`groups`) 같은 다른 페이지에 들어가면 여전히 칙칙한 코드포스의 구형 사이드바와 풀사이즈 화면이 튀어나와 몰입감을 박살 내고 있었습니다.
* **해결 설계:** 컨트롤러 내부에서 레이아웃을 조작하는 시야가 좁은 방식을 폐기합니다. 확장 프로그램이 실행되는 가장 첫 진입점(`main.js` 또는 `theme-init.js`)에서 전역으로 사이드바 요소를 탐색하여 철거하고, CSS(`overrides.css`)를 통해 코드포스의 최상위 컨테이너 아이디(`pageContent`) 자체가 어느 페이지에서든 무조건 백준식 카드 래퍼(Box Shadow, Border Radius, 중앙 정렬) 형태를 띠도록 CSS 스코프를 최상단으로 끌어올려야 합니다.

## 🛠️ 2. 아키텍처 개편 및 조치 지침 (Implementation Guide)

개발자님께서는 아래 지침에 따라 기존 로직을 수정해 주십시오. (절대 코드를 제공하지 않으므로 논리 흐름에 따라 직접 개편을 진행하셔야 합니다.)

### 지침 1. `tierCalculator.js` 정밀도 상향 (유저 티어 세분화)
* 유저 레이팅 변환 함수 내부의 임시 조건문들을 모두 지우십시오.
* 900점 미만부터 3500점 이상까지, 정확히 100점 단위로 끊어지는 명시적인 조건문(IF문) 블록을 작성하십시오. 
* 조건마다 브론즈 5부터 루비 1까지 순차적으로 1:1 매핑되는 정확한 SVG 파일명을 반환하도록 설정하여 404 에러와 변별력 상실을 동시에 해결하십시오.

### 지침 2. 하드코딩 척결 및 설정 객체(Config) 분리
* `problemset.js` 컨트롤러 최상단에 상수(Constant)를 선언하여 화면에 그릴 최대 문제 수(예: 100)를 정의하십시오.
* 배열을 자르는 내장 함수(`slice` 등)에 숫자를 직접 쓰지 말고, 선언해 둔 상수를 변수로 전달하십시오.
* `overrides.css` 최상단 변수 선언부에 레이아웃의 최대 너비(Max Width)를 통제하는 CSS 변수를 추가하고, 이를 활용해 모든 페이지의 폭을 일관성 있게 제어하십시오.

### 지침 3. 글로벌(Global) 백준 레이아웃 강제화
* **JS 조치:** `main.js`의 라우터 분기문이 실행되기 전(최상단)에, DOM에서 사이드바 요소를 찾아 즉시 숨김(Display None) 처리하는 로직을 한 번만 작성하십시오. 이렇게 하면 어느 페이지로 이동하든 사이드바가 일괄 철거됩니다.
* **CSS 조치:** `overrides.css`에서 특정 페이지 클래스(예: `.problemset-mode`)에 종속되어 있던 배경색과 그림자 속성을 빼내어, 모든 페이지가 공유하는 `#pageContent` 메인 래퍼(Wrapper)에 직접 부여하십시오.
* **헤더/푸터 조치:** 모든 페이지에 공통으로 노출되는 하단의 저작권 정보(Footer) 영역의 텍스트 색상과 여백도 백준 스타일로 깔끔하게 정리하여, 페이지 하단부에서 코드포스의 느낌이 새어 나오는 것을 막으십시오.
